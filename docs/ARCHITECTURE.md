## Ключевые архитектурные решения

### Экспозиция и зависимость событий

Экспозиция — это зависимость одного типа событий от другого. При создании типа события можно задать поле `requires_event_type` с типом события-экспозиции.

### Оптимизация критичного пути batch events (Redis-кеш)

Для оптимизации горячего пути (массовая вставка событий) используется кеширование в Redis. Без этого создавалась бы слишком большая нагрузка чтения/записи на таблицу Events в Postgres, так как для каждого события проверяется, не является ли оно экспозицией другого события, которое пришло раньше.

Алгоритм:

1. **Событие не требует экспозиции:** записывается в таблицы `Events` и `EventsRaw` со статусом `received`, кешируется в Redis. Проверяется, есть ли в Redis события, ожидающие пришедшее событие — если есть, их статусы обновляются на `received` и они записываются в `Events`.

2. **Событие требует экспозиции:** проверяется наличие события-экспозиции в Redis. Если есть — пишем в `Events` и `EventsRaw` со статусом `received`. Иначе — отправляется в Redis на ожидание и в `EventsRaw` со статусом `pending`.

Таблица `Events` — для продуктовых метрик, таблица `EventsRaw` — для технических (количество событий, отклоненные, причины).

**Компромиссы:**
- При очистке Redis события навсегда останутся в `pending` если не дождались экспозиции
- Максимальная глубина зависимости — 1 (экспозиция не может требовать экспозицию)
- Поиск по всей таблице `EventsRaw` при обновлении статуса pending-события

Решение выбрано потому что оно разгружает таблицу `Events` и не создаёт нагрузку в виде большого количества чтений по всей таблице, а поиск в Redis занимает значительно меньше времени, чем в Postgres.

### Ограничение нагрузки при подсчёте метрик — Materialized View

Раз в несколько секунд/минут (задаётся в конфиге) строится Materialized View в Postgres с данными таблицы `Events` и связанных таблиц (варианты, типы экспериментов и т.д.). Это сырые данные без агрегации. Агрегация происходит в рантайме при запросе отчёта, что позволяет пересчитывать метрики под любое окно. При этом агрегация в reports не затрагивает таблицу `Events` — чтение происходит из MV.

**Компромиссы:**
- MV перестраивается не постоянно — отчёт может содержать данные с задержкой
- MV считается даже если отчёт не запрашивается (избыточные запросы к БД)

Контролируемые избыточные запросы лучше, чем неконтролируемая нагрузка.

### Stickiness — детерминированная раздача вариантов

Эксперимент раскатывается на заданный процент аудитории. Распределение по вариантам реализовано через hash-based алгоритм: строится SHA-256 хеш на основе `"{subject_id}:{experiment_id}"`, нормируется к 100, после чего субъект попадает в один из бакетов.

Если пользователь не попадает в аудиторию — он получает `default` значение флага и `decision_id = null`. Решение не сохраняется, но результат детерминирован и воспроизводим при повторном запросе.

### Guardrails и Rollback

Раз в N секунд (настраивается) пересчитываются guardrail-метрики для всех вариантов, кроме контрольного. При превышении порога эксперимент ставится на паузу (`PAUSE`) или завершается (`ROLLBACK`).

Трактовка rollback единообразна: эксперимент переходит в `FINISHED` с фиксированием бизнес-решения `ROLLBACK` и описанием. После этого пользователь видит default-значение.

**Компромисс:** контрольный вариант может не совпадать с default-значением (control задаётся вручную при создании). При откате система всегда возвращается к default-значению, а не к control-варианту, чтобы обеспечить предсказуемое поведение.

### Метрики эксперимента

К эксперименту можно прикрепить метрики из каталога метрик трёх типов: `PRIMARY` (основная), `SECONDARY` и `GUARDRAILS`. Guardrail-метрики скрыты из основного отчёта, остальные считаются целиком и для каждого варианта отдельно.
